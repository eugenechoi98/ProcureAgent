# ProcureGuard User Paths

ProcureGuard AI has three user paths. They serve different audiences and should not be described as the same product surface.

## Path A: Manual Audit (Recommended)

**Path A is the primary user entry point.**

Path A is the product-oriented MVP flow:

```text
Manual invoice fields
-> explicit mock PO/GRN context
-> deterministic audit engine
-> AuditReport
```

- Fast.
- Stable.
- No ML dependency.
- No model checkpoint required.
- No GPU required.
- Suitable for clean-clone Quickstart and product-like demos.

This path demonstrates the core procurement audit system: three-way matching, policy checks, duplicate detection, deterministic risk/action, and AuditReport export.

## Path B: AI Extraction (Experimental)

Path B is the research/demo AI vision flow:

```text
Invoice image
-> OCR tokens + bbox
-> LayoutLMv3 field candidates
-> human confirmation
-> deterministic audit engine
-> AuditReport
```

- Uses LayoutLMv3.
- CPU inference is approximately 3 minutes per invoice image.
- Requires a recovered fine-tuned checkpoint, processor, BIO label map, and runtime manifest.
- Produces field candidates, not final audit facts.
- Requires confirmation before fields can enter the deterministic audit engine.

This path is useful for showing the model extraction layer, but it is not the primary product entry point and should not be presented as production-ready live invoice processing.

## Path C: Debug / Developer Mode

Path C is for developers inspecting intermediate artifacts:

```text
Raw candidates
-> confirmation decisions
-> trace view
-> guard / fallback metadata
```

- Exposes raw candidates and trace details.
- Helps debug OCR, LayoutLMv3 reconstruction, field confirmation, Phase 2 audit, and explanation fallback.
- Not intended for normal users.
- Not a payment workflow.
- Not a replacement for the recommended Manual Audit path.

Use this mode to understand why a field was accepted, corrected, rejected, or blocked before reaching the deterministic audit engine.

## Product vs Research Boundary

- Product path: **Path A Manual Audit**.
- Research/demo path: **Path B AI Extraction**.
- Developer diagnostics: **Path C Debug / Developer Mode**.

Across all paths, `risk_level` and `recommended_action` are generated by deterministic rules only. LayoutLMv3 provides candidate fields, and LoRA is limited to guarded explanation rewrite; neither model decides procurement risk or payment action.
